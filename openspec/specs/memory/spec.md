# memory Specification

## Purpose
Give PCC a persistent, vector-searchable memory layer: the user can store facts, notes, and
snippets that are recalled by semantic similarity and surfaced in other plugins (e.g. chat context).
Embeddings are generated by Ollama; storage and retrieval use Qdrant (already in infra).

## Requirements

### Requirement: Memory store endpoint

A new `memory` plugin SHALL expose:

- `POST /api/memory` — store a memory. Body: `{ content: string, tags?: string[] }`.
  Generates an embedding via Ollama (`POST /api/embeddings`, model configurable), upserts to
  Qdrant collection `pcc_memory` (point id = UUID, payload `{ content, tags, createdAt }`).
  Returns `201` with `{ id }`.
- `GET /api/memory?q=<text>&limit=<n>` — recall top-n memories semantically closest to `q`
  (Qdrant vector search). Returns `[{ id, content, tags, createdAt, score }]`.
  When `q` is absent, returns the most-recently-added `limit` memories (Qdrant scroll).
- `DELETE /api/memory/{id}` — delete one memory by id. Returns `204` or `404`.

#### Scenario: Store and recall a fact

- **WHEN** the user stores `"My favourite editor is Neovim"` and later queries `q=text editor`
- **THEN** the memory appears in the top results with a high similarity score

#### Scenario: Empty query returns recent memories

- **WHEN** the user calls `GET /api/memory` with no `q`
- **THEN** the response lists the most recently stored memories, newest first

#### Scenario: Qdrant unreachable

- **WHEN** Qdrant is unreachable
- **THEN** `POST /api/memory` and `GET /api/memory` return `502`

#### Scenario: Ollama embedding model unreachable

- **WHEN** Ollama is unreachable
- **THEN** `POST /api/memory` returns `502` (cannot generate embedding)

### Requirement: Qdrant collection bootstrap

On plugin startup, the `memory` plugin SHALL create the `pcc_memory` Qdrant collection if it does
not exist, using the vector dimension matching the configured embedding model
(`Plugins:Memory:EmbeddingModel`, default `"nomic-embed-text"`). Collection creation failing
(e.g. already exists) SHALL be treated as a no-op.

#### Scenario: Collection created on first boot

- **WHEN** the plugin starts and `pcc_memory` does not exist in Qdrant
- **THEN** the collection is created with the correct vector dimension; subsequent requests succeed

### Requirement: Memory UI

The `memory` plugin SHALL contribute a `/memory` page and a `memory-count` dashboard tile.

The page SHALL show:
- A search bar: typing triggers `GET /api/memory?q=<text>` (debounced 300 ms); clearing shows
  recent memories
- A list of memory cards: content, tags (chips), date, similarity score (when from search),
  a Delete button
- A **New Memory** button opening a modal with a `Textarea` for content and a tag input;
  submits via `createServerFn POST`
- Deletes go through `createServerFn DELETE`

The dashboard tile shows the total memory count.

#### Scenario: Search recalls relevant memories

- **WHEN** the user types in the search bar
- **THEN** the list updates (debounced) to the top-10 most similar memories

### Requirement: Config

```
Plugins:Memory:Enabled           = true
Plugins:Memory:Qdrant:BaseUrl    = "http://qdrant:6333"
Plugins:Memory:EmbeddingModel    = "nomic-embed-text"   # must be pulled in Ollama
Plugins:Memory:Ollama:BaseUrl    = "http://ollama:11434"
Plugins:Memory:DefaultRecallLimit = 10
```

Qdrant and Ollama base URLs default to the compose service names. Missing/unreachable Qdrant → `502`.
