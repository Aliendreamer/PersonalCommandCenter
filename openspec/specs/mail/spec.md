# mail Specification

## Purpose
Add an AI-assisted email client to PCC: fetch emails via IMAP, display threaded conversations,
and use a local Ollama model to summarise threads, draft style-matched replies, auto-tag, and
flag likely spam — all without email leaving the local network.

## Requirements

### Requirement: IMAP sync endpoint

A new `mail` plugin SHALL expose `GET /api/mail/messages?folder=<name>&limit=<n>&offset=<n>`
that connects to the configured IMAP server, fetches message headers (From, To, Subject, Date,
Flags, UID) from `folder` (default `INBOX`) newest-first, and returns them as a JSON array.
Full message body is fetched lazily: `GET /api/mail/messages/{uid}?folder=<name>` returns the
decoded plain-text body (HTML stripped) and attachments list (name + size, no content).

The IMAP connection SHALL be pooled (one persistent connection per folder, idle via IMAP IDLE where
supported). Credentials never leave core-api.

#### Scenario: List inbox

- **WHEN** the user requests `GET /api/mail/messages?folder=INBOX&limit=20`
- **THEN** the response lists up to 20 message headers newest-first

#### Scenario: Fetch message body

- **WHEN** the user requests `GET /api/mail/messages/12345?folder=INBOX`
- **THEN** the response contains the decoded plain-text body of that message

#### Scenario: IMAP unreachable

- **WHEN** the IMAP server is unreachable or credentials are wrong
- **THEN** all `/api/mail/*` endpoints return `502`

### Requirement: Send endpoint

The `mail` plugin SHALL expose `POST /api/mail/send` accepting
`{ to: string[], subject: string, body: string, replyToUid?: number, folder?: string }`.
The endpoint SHALL submit via SMTP and return `201` on success, `400` on missing fields, `502`
on SMTP failure.

#### Scenario: Send a reply

- **WHEN** the user POSTs a valid send request with `replyToUid`
- **THEN** the email is sent via SMTP with correct `In-Reply-To` / `References` headers and `201`
  is returned

### Requirement: AI summarise

The `mail` plugin SHALL expose `POST /api/mail/summarise` accepting `{ uid: number, folder?: string }`.
It SHALL fetch the message body, build a prompt `"Summarise this email in 3 bullet points:\n\n{body}"`,
call Ollama (`/api/generate`, model = `Plugins:Mail:AiModel`), and return `{ summary: string }`.
If Ollama is unreachable the endpoint returns `503` (AI unavailable, not a mail failure).

#### Scenario: Summarise a long email

- **WHEN** the user requests a summary for a long email UID
- **THEN** the response contains a concise bullet-point summary generated by the local model

### Requirement: AI draft reply

The `mail` plugin SHALL expose `POST /api/mail/draft` accepting `{ uid: number, instruction?: string, folder?: string }`.
It SHALL fetch the original body + sender, build a prompt that matches the sender's tone and
incorporates the optional instruction (e.g. "decline politely"), call Ollama, and return
`{ draft: string }`. Returns `503` when Ollama is unreachable.

#### Scenario: Draft a reply

- **WHEN** the user requests a draft with `instruction: "accept and suggest Thursday"`
- **THEN** the response contains a draft reply in a matching tone that agrees and proposes Thursday

### Requirement: Auto-tag

The `mail` plugin SHALL expose `POST /api/mail/tag` accepting `{ uid: number, folder?: string }`.
It SHALL classify the email into one of a configured tag list (`Plugins:Mail:Tags`, default
`["work", "personal", "newsletter", "finance", "travel"]`) via Ollama and return `{ tag: string }`.
Tagging is best-effort; Ollama unavailable returns `503` (not stored).

### Requirement: Mail UI

The `mail` plugin SHALL contribute a `/mail` page with:
- A folder sidebar (INBOX, Sent, Drafts; configurable)
- A message list (subject, sender, date, tag chip, unread bold)
- A reading pane (body, attachment list, **Summarise** button, **Draft Reply** button)
- A compose/reply modal (To, Subject, Body textarea pre-filled from draft)
- Summarise and Draft buttons call their server fns and show inline results; Summarise result
  appears above the body; draft pre-fills the compose modal

Writes (send, tag) go through `createServerFn POST`. Reads use the SSR-BFF loader pattern.

#### Scenario: Summarise in reading pane

- **WHEN** the user opens a message and clicks Summarise
- **THEN** a summary panel appears above the message body within a few seconds

#### Scenario: Draft pre-fills compose modal

- **WHEN** the user clicks Draft Reply
- **THEN** the compose modal opens with a draft body; the user can edit before sending

### Requirement: Config

```
Plugins:Mail:Enabled          = true
Plugins:Mail:Imap:Host        = ""          # required; empty → 502 on all endpoints
Plugins:Mail:Imap:Port        = 993
Plugins:Mail:Imap:Ssl         = true
Plugins:Mail:Imap:Username    = ""          # secret → .env  Plugins__Mail__Imap__Username
Plugins:Mail:Imap:Password    = ""          # secret → .env  Plugins__Mail__Imap__Password
Plugins:Mail:Smtp:Host        = ""
Plugins:Mail:Smtp:Port        = 587
Plugins:Mail:Smtp:Username    = ""          # secret → .env
Plugins:Mail:Smtp:Password    = ""          # secret → .env
Plugins:Mail:AiModel          = "llama3.2"  # must be pulled in Ollama
Plugins:Mail:Ollama:BaseUrl   = "http://ollama:11434"
Plugins:Mail:Tags             = ["work","personal","newsletter","finance","travel"]
```

No IMAP host configured → all endpoints return `502` and the tile shows a degraded state.
AI features (summarise, draft, tag) degrade independently when Ollama is unreachable.
